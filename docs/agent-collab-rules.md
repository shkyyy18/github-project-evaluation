# 子代理协作与升级建议收录规则（全局通用版）

适用范围：shkyyy18 名下全部开源项目（cc-autopilot、kimi-adapter、openbid-intel、mi-fitness-data-bridge、ai-money-lab、llm-video-mcp、github-project-evaluation 及后续新增仓库）。
各仓库的 AGENTS.md 在此基础上叠加本仓定制规则；定制规则与本文件冲突时，以更严格者为准。

## 1. 子代理工作规范

### 1.1 搜索与操作范围

- 只允许读取/修改任务声明中明确列出的仓库和目录；不得顺手翻改其他项目。
- 禁止触碰：用户浏览器及其数据、任何凭证/密钥文件（.env、keyring、token、cookie 等）、工作目录之外的文件。
- 只读探索类任务不得产生任何写操作（包括 `pip install` 之外的临时产物也要说明）。
- 向外发布类动作（git push、开 PR/issue、评论、上传第三方服务、发消息）必须先经用户确认，一次确认只覆盖当次动作。

### 1.2 产出要求

- 所有结论必须基于真实运行的命令输出或真实读到的文件内容，附证据（文件:行号 或命令+输出摘要）；禁止编造测试结果、性能数字、"已通过"结论。
- 没跑过的验证就明说"未验证"，不得把未验证的改动包装成已完成。
- 汇报必须包含遗留项清单：没做完的、不确定的、发现但超出本次任务范围的问题。

## 2. 升级建议收录规则

任何 agent 在工作中发现的改进建议（bug、体验、性能、文档、架构均可），**必须落到文字**，不允许只停留在口头汇报：

1. 写入**该建议所属仓库**的 `AGENTS.md` 的「升级建议 backlog」小节；建议不属于任何仓库时，写入本文件末尾的「全局 backlog」。
2. 每条 backlog 必须包含四个字段：
   - **描述**：问题或改进点是什么，附证据位置（文件:行号）；
   - **发现日期**：YYYY-MM-DD；
   - **来源任务**：哪个任务/哪次工作中发现的；
   - **预估价值/成本**：价值（高/中/低 + 一句话理由）、成本（小/中/大 + 一句话理由）。
3. 写入前先核对现状：已修复的条目标记为「已修复」并注明修复日期，而不是直接删除（保留追溯）。
4. 口头汇报中提到的建议，主代理负责督促落盘；任务验收时检查 backlog 是否更新。

## 3. 升级建议有效性判定与采纳规则（通用框架）

### 3.1 有效性三要件（缺一不可）

1. **可复现 / 有证据**：有具体的文件位置、命令输出、报错信息或实测数据支撑，不是"感觉可以更好"。
2. **与项目定位一致**：符合该仓库 README 声明的定位与边界（如 local-first、零依赖、非官方实验性质），不把项目推向它明确不做的事。
3. **不引入新风险**：不新增安全隐患（凭证、隐私数据暴露面）、不破坏既有兼容性承诺、不增加不可控的外部依赖。

### 3.2 采纳分级

| 级别 | 类别 | 处置 |
| --- | --- | --- |
| 立即做 | bug 修复、安全问题、数据正确性问题 | 尽快单独立项修复，修复后回填 backlog 状态 |
| 排期做 | 体验改进、性能优化、文档完善、重构 | 记入 backlog 排期，每周评审时排序 |
| 记录不做 | 超出项目 scope、与定位冲突、成本远大于价值 | 保留在 backlog 并标注「不采纳」+ 写明理由，防止重复提出 |

### 3.3 评审节奏

- 并入**每周一**的项目组合评分流程（`scripts/score_portfolio.py` 周评）同时进行：过一遍各仓库 AGENTS.md 的 backlog，逐条按 3.1 判定有效性、按 3.2 定级，更新条目状态。
- 每条 backlog 的状态流转：`待评审 → 已采纳（排期/立即） / 不采纳（含理由） / 已修复（含日期）`。
