# 调研操作规程（Research Playbook）

- 版本：v1.0（2026-07-20）
- 来源：从高星调研类开源项目与权威年度调查报告中蒸馏（样本清单见文末附录）
- 触发场景：任何"帮客户/我们调研某个领域、某项业务、某类工具"的任务。**凡涉及真实世界的市场、竞品、规则、趋势，一律先过本规程，再动笔。**

> 核心教训（来自微信分销调研被批事件）：**不要就题答题。先画市场格局，再回答问题。** 漏掉头部玩家（微盟、有赞）的调研，结构写得再细也是废品。

---

## 第 0 步：调研类型判定（动笔前必须写清楚）

客户的一句话需求通常是多种调研的混合体，先拆型，不同类型结构不同：

| 类型 | 典型问法 | 必备环节 |
|---|---|---|
| 市场/进入调研 | "这个业务能不能做" | 格局地图 + 规模/趋势 + 玩家 + 风险 |
| 竞品调研 | "对标谁、怎么选" | 格局地图 + 竞品对比表（统一维度）+ 选型建议 |
| 合规调研 | "这么做会不会违规" | 官方原文 + 处罚案例 + 红线清单 |
| 技术可行性 | "能不能用 X 实现" | 官方能力边界 + 已有实现（开源/SaaS）+ 差距分析 |
| 综合 | 大多数客户问题 | 拆成上面若干子调研，各自给结论 |

**输出物开头必须有一行"调研类型"声明。** 混合型要列出子问题清单，每个子问题注明"回答到什么程度算完"（完成判据）。没有完成判据的调研会无限拖延或提前收工。

---

## 第 1 步：市场格局地图（强制环节，不可跳过）

在回答任何具体问题之前，先回答三个格局问题：**这个领域分哪几类玩家？每类的头部是谁？盘子多大、往哪走？**

格局地图最少要包含：

1. **品类划分**：按商业模式/交付形态分层（如：官方平台原生能力 / 头部 SaaS / 垂直工具 / 服务商代运营 / 开源方案 / 海外对标）。
2. **玩家分层表**：每层列出头部玩家（至少 3 家/层，凑不齐要说明为什么），标注其状态：上市公司 / 融资独角兽 / 头部 SaaS / 平台官方 / 开源项目。
3. **规模与趋势**：市场规模、增速、关键拐点（政策、平台规则大改）。数字必须带日期与来源。
4. **格局对题目结论的影响**：一句话说清"格局如何改变了答案"（如：既然微盟/有赞已把合规分销做成标准品，客户自建的价值就只剩差异化环节）。

### 怎么保证头部玩家找全（具体动作，不是口号）

- 搜"XX 行业 报告 头部 厂商 / 市场份额"，找 1-2 份行业报告（艾瑞、易观、IDC、36氪研究院等）做骨架；
- 查该领域**上市公司名录**（A股/港股/美股招股书、年报里的"竞争对手"章节是免费的竞品清单）；
- 查**融资数据库口径**（IT桔子、Crunchbase）里该赛道的大额融资公司；
- 查**平台生态市场/服务市场榜单**（如微信服务商市场、Shopify App Store 分类排行）；
- **竞品互查**：拿已知的 1-2 家头部，搜"XX 的竞品 / XX vs"，让竞品互相指认；
- 查海外对标（该模式在海外市场谁是头部，国内谁在抄它）。

---

## 第 2 步：竞品覆盖自检清单（交付前逐项打勾）

漏头部玩家是致命伤。交付前对照此表，**任何一项为"否"都必须在报告中显式写出"未覆盖 X 及排除理由"**：

- [ ] 该领域的**上市公司**是否已全部入表？
- [ ] **融资到 B 轮以后/估值独角兽**的公司是否已入表？
- [ ] **头部 SaaS 厂商**（该品类的标准品提供方）是否已入表？
- [ ] **平台官方原生能力**（平台自己下场做的功能）是否已评估？（很多"创业机会"已被官方功能覆盖）
- [ ] **开源/自建方案**是否已评估？
- [ ] **海外对标**是否至少看过一家？
- [ ] 对比维度是否统一（同一张表、同一列定义），还是各写各的？
- [ ] 是否给了**选型建议**（什么类型的客户选什么路径），而不只是罗列？

## 第 3 步：信源分级与交叉验证

### 信源分级表

| 级 | 信源 | 用法 |
|---|---|---|
| S | 官方原文：平台规则原文、监管法规、官方开发者文档、政府公告 | 可直接作为结论依据，引用原文条款 |
| A | 上市公司财报/招股书、权威行业报告、法院判决文书 | 可直接作为结论依据 |
| B | 主流财经/科技媒体、行业协会数据 | 可用，关键事实需第二信源 |
| C | 垂直媒体、从业者深度文、知乎/公众号专业作者 | 可用，必须标注；关键事实必须交叉验证 |
| D | 厂商营销稿、自家产品测评文、软文 | **只用作"该厂商声称什么"，不得作为事实依据** |

### 硬规则

1. **关键事实（结论依赖的每个事实）≥ 2 个独立信源，其中至少 1 个 S/A 级。** 达不到的，正文就地标注"单一信源，未独立核实"。
2. **利益相关声明**：引用厂商/服务商材料时必须标注其利益身份（"销客多为分销工具厂商"），其说法只能代表其立场。
3. **数字三件套**：任何数字必须带 (a) 数值 (b) 截至日期 (c) 来源链接。缺一项的数字不许进正文。
4. **优先引用原文而非转述**：平台规则、法律条文必须引原文条款号，不引"别人对该规则的解读"。
5. **主动找反证**：给结论找至少一个"可能让它不成立"的证据或案例（如合规路径的失败/被处罚案例）。找不到也要写"已检索反证，未发现"。

## 第 4 步：标准报告结构模板

```
# <主题>调研报告
- 日期 / 调研类型 / 委托问题原文 / 完成判据

## 0. 结论摘要（TL;DR）          —— 每条结论可追溯到正文章节
## 1. 市场格局地图              —— 强制：品类划分 + 玩家分层表 + 规模趋势 + 格局对结论的影响
## 2. 需求与规模验证            —— 客户要做的这件事，需求是否真实存在、盘子多大
## 3. 玩家与竞品分析            —— 统一维度对比表 + 选型建议（过什么第 2 步自检清单）
## 4. 规则与红线（如涉及）       —— S 级原文条款 + 处罚案例
## 5. 方案与路径对比            —— 至少 2 条可选路径（如 自建 vs 采购 SaaS vs 混合），含成本区间与适用前提
## 6. 风险提示                  —— 给客户的书面风险清单
## 7. 不知道清单                —— 集中列出：未核实事实 / 数据缺口 / 需要客户提供的信息 / 建议进一步核实什么
## 附. 信源清单                 —— 按 S/A/B/C/D 分级排列
```

单项调研（纯合规、纯可行性）可裁剪，但**第 1 节格局地图和第 7 节不知道清单不可裁剪**。

---

## 第 5 步：输出质量标准（交付前最后一道闸）

1. **可证伪性**：关键结论要附带"失效条件"——什么情况发生时该结论作废（如"若微信小店关闭推客 API，则本方案第 3 步失效"）。写不出失效条件的结论，降级为"观点"并标注。
2. **时效声明**：规则/政策类信息注明"以官方当日口径为准"并给出复检入口；报告标注建议复检周期（平台规则类建议 ≤3 个月）。
3. **"不知道"规则**：核实不了就写"不知道 + 为什么查不到 + 怎么查"，**禁止用 C/D 级信源把不知道的事装作知道**。结尾"不知道清单"非空是正常且健康的；为空反而要警惕。
4. **自检**：第 2 步竞品覆盖清单全部打勾或写明排除理由；数字三件套齐全；每条 TL;DR 能在正文找到支撑。

## 常见翻车点（血泪清单）

1. **就题答题，漏格局**：直接回答"怎么做"，跳过"这个市场长什么样"——微信分销调研漏微盟/有赞就是这个错。→ 强制第 1 步。
2. **只搜中文二手稿**：全网都是互相抄的软文，把营销口径当行业事实。→ 信源分级 + S 级原文优先。
3. **把厂商营销稿当事实**：卖工具的人说"必须买工具"。→ D 级信源规则 + 利益相关声明。
4. **数字不带日期**：规则和市场半年一变，过期数字是误导。→ 数字三件套 + 复检周期。
5. **结论没有失效条件**：客户三个月后拿着报告来问为什么不管用。→ 可证伪性规则。
6. **只给一条路径**：不对比自建/采购/混合，等于替客户做了他没授权的决定。→ 第 5 节路径对比。
7. **不敢写不知道**：硬编一个看似确定的答案。→ 不知道清单制度。
8. **平台官方下场被忽略**：调研半天发现官方免费功能已覆盖。→ 自检清单"平台官方原生能力"一项。
